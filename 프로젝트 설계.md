# SyncShopper 최종 프로젝트 구조 정리

## 1. 프로젝트 개요

**SyncShopper**는 유튜브 영상 시청 중 사용자가 영상을 일시정지하면, 현재 화면 프레임과 자막을 분석하여 영상 속 상품을 탐지하고 관련 쇼핑 상품을 추천하는 서비스이다.

핵심 흐름은 다음과 같다.

```text
유튜브 영상 일시정지
→ 현재 프레임 캡처
→ 전후 자막 수집
→ AI 상품 후보 분석
→ 커머스 API 검색
→ 상품 재랭킹
→ Top 3 추천 상품 노출
```

기존 기획서에서도 핵심 기능은 **일시정지 → 프레임 캡처 → 자막 확보 → 멀티모달 AI 분석 → 커머스 매칭 → 쇼핑 카드 렌더링**으로 정의되어 있다. 

---

# 2. 레포지토리 전략

## 최종 결정: Monorepo

초기 MVP에서는 레포를 여러 개로 나누지 않고 **하나의 레포지토리**에서 관리한다.

```text
sync-shopper
```

## Monorepo를 선택하는 이유

```text
- 1개월 MVP 개발에 적합
- 프론트, 백엔드, AI 서버, 익스텐션을 한 번에 관리 가능
- API 변경 사항을 빠르게 공유 가능
- 기획서, DB, 인프라 문서를 한곳에서 관리 가능
- 팀 프로젝트 발표 시 전체 구조 설명이 쉬움
```

나중에 서비스 규모가 커지면 아래처럼 분리할 수 있다.

```text
sync-shopper-backend
sync-shopper-web
sync-shopper-extension
sync-shopper-ai-server
```

하지만 현재 단계에서는 **하나의 레포 안에서 폴더만 분리**하는 방식이 가장 적합하다.

# 5. 서버 역할 분리

## SpringBoot

SpringBoot는 **메인 서비스 서버**이다.

담당 기능:

```text
- 로그인
- 회원가입
- JWT 인증
- 사용자 관리
- 상품 관리
- 상품 추천 관리
- 관리자 대시보드
- 관리자 유저 관리
- 관리자 게시글 관리
- 관리자 상품 관리
- 관리자 AI 로그 관리
- 커머스 API 연동
- MyBatis DB 저장
```

---

## FastAPI

FastAPI는 **AI 전용 분석 서버**이다.

담당 기능:

```text
- 이미지 분석
- 자막 분석
- 상품 후보 추출
- 객체 후보 판단
- 브랜드 / 모델명 추론
- 검색 쿼리 생성
- 이미지 임베딩
- 유사도 계산
- confidence 계산
```

FastAPI를 분리하는 이유는 AI 기능 확장성 때문이다.

```text
- Python 기반 AI 라이브러리 사용이 쉬움
- YOLO, OpenCV, CLIP, Embedding 처리에 유리함
- AI 실험과 모델 교체가 쉬움
- SpringBoot의 서비스 로직과 AI 로직을 분리할 수 있음
```

```text
- JWT 토큰 발급
- 일반 사용자 / 관리자 권한 분기
```


```text

- 크롬 익스텐션 설치 유도
- 인기 추천 상품
- 최근 본 상품
- 관심 카테고리 기반 추천 상품
```

```text

- 카테고리
- 가격
- 리뷰 수
- 평점
- 추천 이유
- 구매하러 가기 버튼
- 비슷한 상품
```

### 상품추천페이지


# 8. DAO와 Mapper 구조

본 프로젝트에서는 별도 `dao` 패키지를 두지 않는다.

MyBatis에서는 `Mapper`가 사실상 DAO 역할을 한다.

```text
Controller
→ Service
→ Mapper
→ Mapper XML
→ DB
```

예시:

```text
RecommendationController
→ RecommendationService
→ RecommendationMapper
→ RecommendationMapper.xml
→ recommendations table
```

따라서 최종 구조는 다음과 같다.

```text
controller
service
mapper
domain
dto
external
config
security
common
```

# 9. AI 기능 위치

AI 기능은 한 곳에 몰아두지 않고 역할에 따라 나눈다.

## SpringBoot 쪽

```text
backend/src/main/java/com/syncshopper/external/ai
├─ AiServerClient.java
└─ FastApiAiServerClient.java
```

SpringBoot는 AI를 직접 분석하지 않고, FastAPI AI 서버를 호출한다.

역할:


# 13. 핵심 처리 흐름

## 영상 기반 상품 추천 흐름

```text
1. 사용자가 유튜브 영상을 일시정지한다.
2. 크롬 익스텐션이 현재 프레임을 캡처한다.
3. 크롬 익스텐션이 전후 자막을 수집한다.
4. SpringBoot의 /api/detections로 요청한다.
5. SpringBoot가 FastAPI의 /ai/detect를 호출한다.
6. FastAPI가 이미지와 자막을 분석한다.
7. FastAPI가 상품 후보, 카테고리, 브랜드, 모델명, 검색 쿼리, confidence를 반환한다.
8. SpringBoot가 커머스 API로 상품을 검색한다.
9. RerankingService가 상품 점수를 계산한다.
10. RecommendationService가 Top 3 상품을 확정한다.
11. MyBatis로 분석 결과와 추천 결과를 저장한다.
12. Vue 또는 크롬 익스텐션에 추천 상품을 반환한다.
```

---

## 상품추천페이지 흐름

```text
1. 사용자가 상품추천페이지에 접속한다.
2. Vue가 GET /api/recommendations를 호출한다.
3. SpringBoot가 사용자 관심 카테고리와 행동 로그를 조회한다.
4. RecommendationService가 추천 후보 상품을 가져온다.
5. RerankingService가 점수를 계산한다.
6. 추천 상품 목록을 반환한다.
7. Vue가 추천 상품 카드를 출력한다.
```

# 14. 추천 점수 계산 기준

초기 MVP에서는 복잡한 AI 개인화보다 **규칙 기반 재랭킹**을 우선 적용한다.

```text
최종 추천 점수 =
AI confidence
+ 카테고리 일치도
+ 브랜드 일치도
+ 가격대 적합도
+ 리뷰 수
+ 평점
+ 사용자 클릭 기록
+ 사용자 저장 기록
- 사용자 무시 기록
```

이후 고도화 단계에서 FastAPI의 embedding, similarity 기능을 붙인다.

```text
사용자 취향 벡터
+ 상품 벡터
+ 현재 영상 상품 후보 벡터
→ 유사도 기반 개인화 추천
```

단, 사용자의 브라우저 전체 검색이력이나 유튜브 전체 시청기록 원본을 저장하는 방식은 지양하고, **서비스 내부 행동 기록과 요약된 취향 프로필**을 활용하는 방식이 적합하다.
